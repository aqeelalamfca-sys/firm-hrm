Critical improvements required
1) True OCR is not actually implemented yet

At present, scanned PDFs and images are not being fully OCR-read in a reliable audit-grade way.

Current issue:

PDF text extraction uses normal PDF parsing
image files are mostly treated as placeholders for vision analysis
scanned PDFs are not being converted page-by-page through a robust OCR pipeline

Required fix:

add a real OCR layer for scanned PDF pages and images
process every PDF page individually
store page-wise extracted text and confidence
mark source as:
native text PDF
OCR PDF
image OCR
user-uploaded Excel native data

This is the biggest gap.

2) Upload rules are too broad

Your required workflow is specific:

Financial Statements → Excel
Sales Tax Return → PDF

But current upload accepts many file types.

Required fix:

make document categories strict:
Financial Statements / TB / GL / schedules → Excel only
Sales tax return / notices / annexures → PDF only
allow override only through admin setting
show upload validation before moving forward
3) Extraction should come before variable editing

Right now the flow places “Engagement Variables” before the extraction result is fully reviewed.

Required fix:
Use this sequence:

Upload files
AI scan + OCR + extract
Show extracted arranged data
Auto-fill variables
User edits variables if wrong
Lock approved variables
Start generation step-by-step

This will match your intended process much better.

4) “Arranged data” layer is missing

You asked for conversion into arranged data first. That layer should exist separately before TB/GL generation.

Required fix:
Create a staging area with these tabs:

Entity Profile
Reporting Metadata
FS Line Items
Prior Year Comparatives
Sales Tax Data
Tax Period Summary
Notes / Schedules
Exceptions & Missing Data
Assumptions Register
Extraction Log

Each row should carry:

source file
sheet/page no.
field name
extracted value
confidence %
editable override
final approved value

Without this structured staging layer, the system remains too prompt-driven.

5) TB generation should be rule-based first, AI-assisted second

At present, the system uses AI heavily and even auto-balances through retained earnings adjustments if needed. That is risky.

Required fix:
TB generation should follow this order:

first derive from structured FS data using deterministic rules
then use AI only for mapping and missing account suggestions
never silently force-balance through retained earnings
any balancing adjustment must go to:
Exception Log
Proposed Adjustment Sheet
User Review Required

Mandatory controls:

TB total debits = total credits
FS head mapping total = FS amount
unmatched balances highlighted
balancing difference cannot be hidden
6) GL generation needs a stronger audit-control engine

Current GL generation is still AI prompt-led. That may produce “logical-looking” ledgers, but not always controlled ledgers.

Required fix:
Build GL using a rules engine with these controls:

opening balance carried from prior year / approved opening
monthly transaction spread based on account nature
voucher sequence continuity
debit/credit logic per account type
period-end closing logic
sales tax linkage where applicable
withholding tax linkage where applicable
closing balance must match TB exactly
total debits and credits per account must match TB turnover exactly

Also add:

ledger generation rationale
transaction count logic note
synthetic ledger flag where reconstructed by AI
